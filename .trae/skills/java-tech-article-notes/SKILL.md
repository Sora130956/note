# Java 技术文章笔记整理

## 目标

将内置浏览器中正在阅读的技术文章，整理为面向 Java 后端开发者的高价值 Markdown 学习笔记。在保证关键信息不丢失的前提下尽量精简。

## 何时调用

当用户提出以下需求时调用本技能：

- 总结内置浏览器当前文章
- 提炼技术要点并形成学习笔记
- 将笔记保存为当前工程内的 Markdown
- 按技术类型分类存放
- 同主题文档存在则追加，不存在则新建

## 输出要求

- 内容尽量精简，但不能丢失关键信息。
- 聚焦 Java 后端可落地结论。
- 保留核心事实：概念、架构、API、约束、性能、陷阱、最佳实践。
- 以 `.md` 保存到当前工程。
- 按技术类型放入分类目录。
- 同类/同主题已存在则追加。
- 不存在则新建文档。
- 笔记默认使用中文（标题、解释、结论）。仅在必要时保留英文术语。

## 术语处理规则

- 对“重要但生涩/非通用”的概念，必须给出 1-3 行中文白话解释。
- 必要时可补一个最小示例帮助理解。
- 对“不重要且生涩”的概念，直接舍弃，避免干扰主线。
- 关键术语首次出现采用“英文术语 + 中文解释”，例如：`Function Calling（让模型按固定参数格式请求你调用外部函数）`。
- 避免堆砌未解释术语；无法简明解释且非核心时应删除。
- “关键概念解释（必要）”应放在文档最前面（标题之后优先展示），再进入摘要与核心要点。
- 对于文章中同时出现的两个或多个相关核心概念，应在“关键概念解释”部分分别用 1-2 句话简洁定义，并在后续的“核心要点”中明确它们之间的关系（相似点、不同点、互补性等）。

## 执行流程

1. 获取内置浏览器中的文章内容。
2. 识别技术分类，例如：
   - `java`, `spring`, `database`, `redis`, `mq`, `kafka`, `rpc`, `jvm`, `observability`, `devops`, `architecture`, `security`
3. 识别具体主题并确定文件名，例如：
   - `spring-transaction-propagation`
   - `redis-cache-penetration-breakdown-avalanche`
4. 在项目中搜索同类笔记：
   - 优先复用已有分类目录。
   - 根据文件名与标题判断是否同主题。
5. 决定写入方式：
   - 同主题存在：追加新小节（含日期与来源）。
   - 同主题不存在：新建 Markdown 文档。
6. 按模板写入精简结构化内容。
7. 检查可读性，去重并消除无意义重复。

## 目录与文件规则

- 根目录：当前项目目录。
- 分类目录：`<project>/<category>/`
- 文件名：默认使用英文 kebab-case，例如 `redis-expire-strategy.md`
- 若用户要求中文文件名，按用户要求执行。
- 不覆盖无关历史内容。

## 笔记模板

按以下结构组织：

```markdown
# <Topic Title>

## 关键概念解释（必要）
- 仅解释“重要但生涩”的概念（每个 1-3 行）
- 非重要生涩概念可省略

## 摘要
- 2-4 bullets，概括文章核心价值与适用场景

## 核心要点
- 按主题列出关键机制/原理（精简但完整）

## 关键细节与易错点
- 参数、边界条件、版本差异、常见误区

## Java 后端落地建议（可选）
- 仅保留能讲清楚、可执行的建议
- 抽象口号式描述且无解释时直接删除

## 示例（可选）
- 简短代码或配置片段（仅保留关键部分）

## 延伸阅读（可选）
- 相关概念或下一步学习方向

## 来源
- 标题：
- 链接：
- 记录时间：YYYY-MM-DD
```

## 精简规则

- 删除叙事性文字与重复表达。
- 合并等价观点，避免同义反复。
- 仅保留必要术语与准确约束条件。
- 有数值阈值、复杂度、压测数据时优先保留。
- 保留术语时至少补一句上下文解释。
- 无法解释清楚的抽象实现口号直接删除。

## 质量检查清单

保存前确认：

- 重要信息已保留。
- 相比原文明显更精简。
- 分类正确。
- 追加/新建逻辑正确。
- Markdown 标题层级清晰一致。
- 重要生涩术语已中文解释清楚。
- 不重要生涩术语已删除。
- “关键概念解释（必要）”已位于文档前部（标题后优先）。
- 相关概念之间的关系（相似点、不同点、互补性）已明确阐述。
